Encrypted keystore

        
      

      
A built-in, encrypted place to keep signing keys and secrets
         when you don't run an external credential manager (OpenBao / CyberArk).
         Secrets are stored with envelope encryption — only
         ciphertext is ever written to the database. The keystore boots
         sealed; an admin unseals it after each restart, so a stolen disk
         or database backup reveals nothing. Choose Signing / CA →
         key storage → Encrypted keystore to route generated keys here.

Issuance policy

        
      

      
Guardrails enforced at issue time: allowed CA backends and
         cert types, key type + minimum size, validity cap, SAN/CN allow/deny
         globs, and required subject fields. Policies layer
         global → tenant → template and can only ever tighten. A
         request that violates the effective policy is refused before any CA is
         touched.

      


      
      
When off (default), a violating externally-submitted CSR is
         still accepted and the requester sees the violations as a warning; the
         authoritative refusal happens at issue time. When on, submission is
         rejected immediately.

      User Guide
      
      
      
    
      
    
      
      
      
      
      
        
      

          
      
            
      Welcome to Certheim
      
            
      Certheim is where your team requests, signs, issues, delivers,
               tracks, and renews TLS/PKI certificates — without juggling spreadsheets,
               shared inboxes, or manual OpenSSL commands.
      
            
      The app has two areas, switched from the header:
      
            
        
              
      • Dashboard — day-to-day work: create requests, track
                  jobs, download issued certificates, and watch expirations.
        • 
              
      • Admin — configuration: users, authentication,
                  templates, signing backends, notifications, and the audit log
                  (visible only to administrators).
        • 
            
      
            
      This guide is context-aware. Open it with the
               Guide button in the header and it lands on the page for whatever
               you're currently looking at. Use the contents on the left, the
               Previous / Next buttons, or the ← → arrow keys to move around;
               press Esc to close.
      
            
      
               You're running the Community edition — the free core for
               requesting, signing, tracking, and renewing certificates. The
               Commercial and Government editions add
               single sign-on, an issuance policy engine, additional signing backends,
               certificate inventory, and more.
      
            
      
               You're running the Commercial edition. Alongside the core
               workflow, your administrators can configure single sign-on (OIDC / SAML /
               SCIM), the issuance policy engine, multiple signing backends, automated
               delivery, certificate inventory and CT monitoring, and role-based access —
               each documented in the Administration section of this guide.
      
            
      
               You're running the Government edition. It includes everything
               in Commercial plus the public-sector suite: CAC / PIV authentication,
               public-sector CSR profiles, tamper-evident (WORM) audit, FIPS & HSM key
               ceremonies, separation of duties and dual control, Federal PKI trust anchors,
               NIST 800-53 / OSCAL compliance evidence, STIG/SCAP self-assessment, CNSA 2.0
               and post-quantum migration, and air-gapped CRL export — each documented in the
               Administration and Government sections of this guide.
      
          
      

          
      
            
      Create requests
      
            
      Start a new certificate request here. There are two ways in:
      
            
        
              
      • Generate — the app creates the private key and CSR for
                  you from a template. The key stays on the server only as long as needed
                  to issue, and you download the key + certificate when it's ready.
        • 
              
      • External submit — you already generated a key and CSR
                  elsewhere; paste the CSR and the app routes it for signing. Your private
                  key never leaves your machine.
        • 
            
      
            
      Pick a template first — it sets the certificate's purpose,
               allowed names, validity, and which CA signs it. Then fill in the hostname /
               common name and any additional Subject Alternative Names (SANs).
               Optionally assign the request to one of your groups so teammates
               can see and manage it.
      
            
      Tip: save your notification email once in
               Settings and it pre-fills on every request, so you're told when a
               certificate is issued.
      
          
      

          
      
            
      Jobs
      
            
      Every request becomes a job you can track through its lifecycle:
      
            
        
              
      • Pending — awaiting approval / signing.
        • 
              
      • Issued — signed and ready; download the certificate and
                  its chain (and the key, for Generate jobs).
        • 
              
      • Error — signing failed; open the job to see why.
        • 
            
      
            
      Open a job to see its full detail: the subject, SANs, serial, the CSR, and
               the issued certificate. Depending on your role and the template you may also
               see actions to cancel a pending request, revoke
               an issued certificate, or renew one that's nearing expiry.
      
            
      Jobs assigned to a group are visible to that group's members; personal jobs
               are visible only to you and administrators.
      
          
      

          
      
            
      Signing queue
      
            
      Visible to signers and administrators. This is the approval queue:
               pending requests waiting for a human to authorize issuance.
      
            
        
              
      • Review the request — subject, SANs, requester, and template policy.
        • 
              
      • Choose Approve & sign to issue it against the
                  template's configured CA backend (e.g. OpenBao, ACME, or an enterprise CA).
        • 
              
      • The signing CA key never sits on the app server — the dashboard only asks
                  the backend to sign, and every approval is written to the audit log.
        • 
            
      
            
      Templates can also be set to auto-sign, in which case matching
               requests skip this queue and issue immediately.
      
          
      

          
      
            
      Fleet certs
      
            
      An inventory of the certificates you and your groups are responsible for —
               both ones issued here and any you're tracking. Use it to stay ahead of
               expirations.
      
            
        
              
      • See each certificate's subject, issuer, and days until expiry,
                  with near-expiry items highlighted.
        • 
              
      • Renew issued certificates directly when a template supports it, or let
                  automated renewal handle them in the background.
        • 
            
      
          
      

          
      
            
      My groups
      
            
      Groups let a team share certificate requests. Jobs assigned to a group are
               visible to every member, so coverage doesn't depend on one person.
      
            
        
              
      • See the groups you belong to and their members.
        • 
              
      • When creating a request, pick a group in the assignment dropdown to share
                  it; leave it blank for a personal job.
        • 
            
      
            
      Administrators create groups and manage membership from
               Admin → Groups.
      
          
      

          
      
            
      Templates
      
            
      A read-only view of the certificate templates available to you. A
               template is a reusable recipe that decides:
      
            
        
              
      • the certificate's purpose (server, client, etc.) and key type;
        • 
              
      • the allowed names / domains and validity period;
        • 
              
      • which CA backend signs it and whether issuance needs
                  approval or is automatic.
        • 
            
      
            
      Pick the template that matches what you're requesting on the
               Create requests page. Administrators define templates under
               Admin → Templates.
      
          
      

          
      
            
      Admin · Overview
      
            
      The administrator's home: a snapshot of system status and recent activity —
               job counts by state, user and group totals, and any items needing attention.
      
            
      Use the Admin navigation to configure the rest of the system. A good
               first-run order on a new deployment: CSR Subject → Templates
               → Signing / CA → Authentication → Users.
      
          
      

          
      
            
      Admin · Users
      
            
      Manage who can use the dashboard and what they can do.
      
            
        
              
      • Roles: grant admin (full configuration) or
                  signer (may approve & sign in the queue).
        • 
              
      • Status: activate, deactivate, or approve users who
                  registered and are pending.
        • 
              
      • In local-auth mode users have a first.last username; with
                  CAC/mTLS they're identified by their certificate DN.
        • 
            
      
          
      

          
      
            
      Admin · Authentication
      
            
      Controls how people sign in. A host runs one mode:
      
            
        
              
      • local — username + password, with optional
                  self-registration.
        • 
              
      • mTLS / CAC — client-certificate (smart-card) login.
        • 
            
      
            
      For self-registration you can require admin approval and restrict sign-ups to
               one or more trusted email domains (comma-separated, e.g.
               example.com, mail.mil). You can also set a
               login banner (a consent/notice users must accept), choosing a
               preset or custom text.
      
            
      Caution: don't switch to mTLS until CAC login is proven
               working, or you can lock everyone out — the app requires an explicit confirm.
      
          
      

          
      
            
      Admin · Groups
      
            
      Create groups and assign members so teams can share certificate requests.
      
            
        
              
      • Add or remove members and set their role within the group.
        • 
              
      • Members then see group-assigned jobs on their Dashboard and can pick the
                  group when creating requests.
        • 
            
      
          
      

          
      
            
      Admin · Roles & permissions
      
            
      Shape access control to your organization: define your own roles
               as bundles of permissions, then grant them to people two ways.
      
            
        
              
      • The permission catalog — every action Certheim gates
                  (view inventory, request, approve & sign, revoke, manage templates,
                  view the audit log, manage settings, and more), each with a plain-language
                  description. This is the vocabulary your roles are built from.
        • 
              
      • Roles — start from the four built-ins (admin, operator,
                  auditor, member) or create your own, e.g. PKI Operators
                  (request + approve & sign, no revoke) or Auditors
                  (read inventory and the audit log, change nothing). Clone an existing
                  role to start close; a role can't be deleted while it's still in use.
        • 
              
      • Assign to a user — pick a role in the user table. That
                  user then holds exactly that role's permissions.
        • 
              
      • Attach to a group — every member of the group holds the
                  attached roles' permissions while they're a member. Onboarding a whole
                  team becomes one assignment instead of many.
        • 
            
      
            
      A user's effective permissions are the union of their
               own role plus every role attached to their groups — access only ever adds up.
               Use the Effective permissions lookup to see exactly what someone can
               do and why (which role granted it, via which group).
      
            
      Admins are always full-access — the admin flag sits
               above roles and can't be locked out. A few sensitive areas (license, database,
               the local-CA key ceremony, RBAC management itself) stay admin-only by design.
               Role enforcement is a licensed feature (Commercial and above); without a license
               everything falls back to admin-only, exactly as before.
      
          
      

          
      
            
      Admin · Templates
      
            
      Define the certificate templates users request against. Each
               template sets the policy and the signing path:
      
            
        
              
      • Subject & SAN policy — allowed names/domains, key
                  type, validity (TTL).
        • 
              
      • Signing backend — which CA issues it (manual upload,
                  OpenBao, ACME, or an enterprise CA), plus the backend's role/options.
        • 
              
      • Approval — require a signer's Approve & sign,
                  or auto-sign on request.
        • 
              
      • Auto-renew — let the background renewer reissue before
                  expiry.
        • 
            
      
            
      Only one signing producer runs per template, so a job is never signed twice.

                 hidden. Pick this when your signing process must stay fully out-of-band.
      
            
      

            
      
              
      Local CA (built-in) Commercial / Government

                 private key is envelope-encrypted in the built-in sealed keystore
                 and CSRs are signed locally.

                    import, or sign while the keystore is sealed.

                    Generate a new CA (CN required; EC P-256/P-384 or RSA;
                    validity in days) or Import an existing CA (cert + private
                    key PEM — the key is validated against the cert and re-encrypted into the
                    keystore, so its old passphrase is not kept).
        3. 
                
      3. Download the CA certificate and distribute it to the systems
                    that must trust what you issue (the Trust Store page can push bundles to
                    fleet hosts).
        4. 
                
      4. Revocation is built in: revoke from the job, then Download CRL
                    here and publish it wherever your clients check.
        5. 
              
      
              
      Generating a new CA replaces the previous one, and its issued/revoked index
                 starts fresh. Prefer the HSM backend if you have hardware — a PKCS#11 key is
                 stronger than any software keystore.

                 the Local CA panel and newly issued certs carry AIA/CDP pointers to
                 <base>/ocsp and <base>/crl — Certheim
                 answers OCSP checks and serves the CRL itself.
                 Post-quantum: on OpenSSL 3.5+ hosts the key-type
                 picker also offers ML-DSA-65/87, for a CNSA 2.0-aligned
                 quantum-resistant CA.

      Admin · Delivery destinations
      
            
      After a certificate is issued, Certheim can push it (and, per the template's
               key mode, its private key) to where it's used. Delivery is set
               per template (Admin → Templates): choose the
               delivery backend, a delivery target, and an optional
               reload command. Failed deliveries retry with backoff and raise a
               job.delivery_failed event before being abandoned. Destination
               credentials live in OpenBao KV, one secret per target — never in the
               dashboard database.
      
            
      
              
              
            
      

            
      
              
      OpenBao / Vault KV
      
              
      Writes the issued bundle to KV v2 at
                 <kv mount>/<base>/<target host>, so
                 consumers (agents, CI, ESO) fetch it from the vault instead of from
                 Certheim. Set the KV mount in the delivery settings; the app's own
                 OpenBao credential needs write access to that path. Good default when the
                 fleet already reads secrets from OpenBao.
      
            
      

            
      
              
      SSH push (Linux fleet)
      
              
        
                
      1. Create a delivery user on the destination host and authorise a key for
                    it.
        2. 
                
      2. Store the credential in OpenBao at
                    secret/csr-delivery-ssh/<host> with fields
                    username, private_key, optional
                    port.
        3. 
                
      3. On the template: backend ssh, target = the destination
                    path, and an optional reload command (e.g.
                    systemctl reload nginx) run after the copy.
        4. 
              
      
              
      Whether the private key ships too follows the template's key mode — a
                 cert-only push is the safer default when the host generated its own CSR.

                 short-lived (delivery_pull_ttl, default 1 hour) and
                 consumed on fetch (delivery_pull_max_uses, default 1). The
                 token reaches the requester via the job's notification channel. Right
                 choice when Certheim has no route or credential to the destination.

      Admin · Enrollment & signing services
      
            
      Standards-based endpoints that let devices and toolchains use Certheim
               directly — enrollment over EST or SCEP, plus code-signing and RFC 3161
               timestamping. All of them issue through the signing backend you configured
               under Signing / CA.
      
            
      
              
              
            
      

            
      
              
      EST enrollment (RFC 7030)
      
              
        
                
      1. Enable EST on this panel and set the shared enrollment credential
                    (user + secret). Clients authenticate with HTTP Basic over TLS.
        2. 
                
      2. Endpoints: /.well-known/est/cacerts (CA chain, no auth),
                    simpleenroll / simplereenroll (POST the
                    base64 DER PKCS#10, Content-Type: application/pkcs10),
                    and csrattrs.
        3. 
                
      3. Client check without an EST client:
        
                    openssl req -new -key k.pem -subj /CN=dev1 -outform DER | base64 |
                    curl -u est:SECRET -H "Content-Type: application/pkcs10"
                    --data-binary @- https://HOST/.well-known/est/simpleenroll
        4. 
              
      
            
      

            
      
              
      SCEP enrollment (RFC 8894)
      
              
        
                
      1. Enable SCEP; Certheim generates an RA keypair in-app —
                    the RA private key is stored in the sealed keystore when it's unsealed.
                    Optionally require a challenge password.
        2. 
                
      2. Endpoint: /scep (also answers
                    /scep/pkiclient.exe) with the standard
                    GetCACaps / GetCACert /
                    PKIOperation operations.
        3. 
                
      3. Works with sscep, certmonger, and network gear that only speaks SCEP.
                    sscep example:
                    sscep enroll -u https://HOST/scep -k k.pem -r req.csr
                    -c ca.crt -l cert.crt
        4. 
              
      
            
      

            
      
              
      Code-signing (CMS)
      
              
        
                
      1. Issue or import a signing certificate, then configure its key reference
                    on this panel.
        2. 
                
      2. Sign artifacts from the panel or the admin API
                    (POST /api/admin/codesign/sign) — output is a detached CMS
                    signature.
        3. 
                
      3. Verify:
                    openssl cms -verify -binary -inform PEM -in sig.pem -content
                    artifact -CAfile chain.pem -purpose any — note
                    -purpose any: openssl's default purpose check rejects
                    codeSigning-EKU certs.
        4. 
              
      
            
      

            
      
              
      Timestamping (RFC 3161 TSA)
      
              
        
                
      1. Enable the TSA and configure its signing key; the policy OID defaults to
                    the Certheim policy (1.3.6.1.4.1.55555.3.1) and serials are
                    issued from a monotonic counter.
        2. 
                
      2. Clients POST a timestamp query to /tsa:
        
                    openssl ts -query -data file.bin -sha256 -cert | curl
                    -H "Content-Type: application/timestamp-query" --data-binary @-
                    https://HOST/tsa -o file.tsr
        3. 
                
      3. Verify: openssl ts -verify -data file.bin -in file.tsr
                    -CAfile chain.pem
        4. 
              
      
            
      

            
      
              
      CMP enrollment (RFC 4210)
      
              
      The enrollment protocol of the EJBCA / telecom (3GPP) world. Certheim
                 implements the interoperable bootstrap subset: p10cr
                 (a PKCS#10 inside a CMP envelope) protected by the shared-secret
                 password-based MAC, answered with cp; plus
                 certConf/pkiconf.
      
              
        
                
      1. Enable CMP and set the shared secret (12+ characters) —
                    clients prove possession via the RFC 4210 password-based MAC, so the
                    secret never crosses the wire.
        2. 
                
      2. Endpoint: POST /cmp
                    (Content-Type: application/pkixcmp). Issuance goes through
                    the configured signing backend, same as EST/SCEP.
        3. 
                
      3. openssl 3.x client example:
                    openssl cmp -cmd p10cr -server https://HOST/cmp -csr req.csr
                    -secret pass:SECRET -ref certheim -certout cert.pem
        4. 
              
      
            
      

            
      
              
      S/MIME user certificates
      
              
      Personal certificates for signing/encrypting email — the one certificate
                 category that's about people, not servers. Issued by the
                 Local CA under an emailProtection profile.
      
              
        
                
      1. Prerequisites: Local CA configured, sealed keystore unsealed.
        2. 
                
      2. Issue via the admin API: POST /api/admin/smime/issue with
                    {name, email, days, passphrase} — the response is a
                    passphrase-protected PKCS#12 the user imports into
                    Outlook / Thunderbird / Apple Mail.
        3. 
                
      3. The private key is generated for this call only and never stored —
                    the returned PKCS#12 is the only copy. Lost file or forgotten
                    passphrase: re-issue, and revoke the old certificate.
        4. 
                
      4. Distribute the Local CA certificate to recipients' trust stores so
                    signatures validate (Trust Store page).
        5. 
              
      
            
      

            
      Reverse-proxy note: these services live at root-level paths —
               /.well-known/est/, /scep, /tsa,
               /cmp, and the Local CA's /ocsp + /crl.
               If a proxy in front of Certheim only forwards the app path, add forwarding
               rules for these, or devices will get the proxy's 404.
      
          
      

          
      
            
      Admin · Encrypted keystore
      
            
      A built-in sealed keystore for deployments without OpenBao/CyberArk: secrets
               are envelope-encrypted (a per-secret data key wrapped by a master key that
               only exists unwrapped while the store is unsealed). It protects
               generated private keys (key storage mode sealed), the Local CA key,
               the SCEP RA key, and the code-signing/TSA keys.
      
            
      
              
              
            
      

            
      
              
      Choosing a protection mode (one-time initialisation)
      
              
        
                
      • Passphrase — one admin passphrase unseals the store; a
                    one-time recovery code is shown at initialisation as the
                    fallback. Simplest; right for a single-admin deployment.
        • 
                
      • Shamir shares — the master key splits into
                    n shares of which any k unseal (e.g. 5 shares,
                    3 required). Distribute shares to separate custodians; no single person
                    can unseal. Right for separation-of-duties requirements.
        • 
              
      
              
      Record the recovery material immediately. It is shown
                 once. Lose the passphrase/recovery code (or more than n−k shares) and the
                 sealed contents are unrecoverable by design — there is no back door.
      
            
      

            
      
              
      Choosing a cipher (security scale)
      
              
        
                
      1. AES-256-GCM, in-process — most secure; key material
                    never leaves the Python process. Uses the bundled
                    cryptography package.
        2. 
                
      2. AES-256-CBC + HMAC, in-process — middle option; same
                    in-process property with a more conservative construction.
        3. 
                
      3. openssl subprocess — dependency-free fallback that
                    shells out to the system openssl; works on any host, including minimal
                    installs where cryptography is unavailable.
        4. 
              
      
              
      Pick at initialisation. Stored blobs are self-describing, so existing
                 secrets keep decrypting if the deployment's cipher availability changes.
      
            
      

            
      
              
      Day-to-day: unseal, seal, secrets
      
              
        
                
      • The store starts sealed after every service restart —
                    unseal it here (passphrase, recovery code, or k shares). Until then,
                    anything needing a sealed secret (Local CA signing, SCEP RA, sealed key
                    storage) refuses cleanly rather than failing oddly.
        • 
                
      • Unsealing once covers all workers; you don't unseal per process.
        • 
                
      • Seal manually before maintenance you don't trust — it
                    drops the master key from memory immediately.
        • 
                
      • The secrets list shows names and timestamps only; values are never
                    displayed back.
        • 
              
      
            
      

            
      
              
      Backup & restore
      
              
        
                
      • Export backup produces an encrypted bundle under a
                    passphrase you choose at export; store it offline. The bundle is
                    self-contained — restorable on a fresh install even before that install
                    has cryptography available.
        • 
                
      • Restore imports the bundle (with an overwrite toggle
                    for name collisions).
        • 
                
      • Re-export after adding significant secrets (a new Local CA, a new
                    signing key) — the backup is a snapshot, not a live mirror.
        • 
              
      
            
      
          
      

          
      
            
      Admin · Tenants Commercial / Government
      
            
      Hard-isolated tenants give each business unit or customer its own data store
               under one Certheim installation. This page is the tenant registry: create a
               tenant, activate it, and its members work in a fully separate space — jobs,
               templates, settings and inventory never cross tenants.
      
            
      Creating a tenant
      
            
      Add a tenant with a short slug (it becomes part of the data-store identity, so
               pick it deliberately — slugs don't rename). New tenants start empty: bootstrap
               an admin for the tenant the same way you bootstrapped the default one.
      
            
      Gotchas
      
            
      Everything before your first tenant lives in the default tenant and
               stays there — creating tenants doesn't migrate existing data. Cross-tenant
               reporting is deliberately absent; that isolation is the feature.
      
          
      

          
      
            
      Admin · Provisioning (SCIM 2.0) Commercial / Government
      
            
      Let your identity provider (Okta, Entra ID, Ping, …) create, update and
               deactivate Certheim users and groups automatically. Certheim exposes a standard
               SCIM 2.0 endpoint; the IdP is the source of truth and pushes changes as they
               happen — no more manual offboarding.
      
            
      Setup
      
            
      Enable SCIM here and copy the base URL plus the generated bearer token into your
               IdP's SCIM connector. Assign the app to the users/groups you want provisioned.
               The token is shown once — rotate it from this page if it leaks, then update the
               IdP.
      
            
      Gotchas
      
            
      SCIM manages the LIFECYCLE (who exists, group membership); it does not sign
               anyone in — pair it with SSO. Deactivation via SCIM disables the user but keeps
               their history for audit. Locally-created users are left alone unless the IdP
               claims the same identity.
      
          
      

          
      
            
      Admin · Observability Commercial / Government
      
            
      Expose a Prometheus /metrics endpoint with the numbers an on-call
               actually wants: certificate counts by status, expiry buckets, delivery states,
               pending approvals. Scrape it into Prometheus/Grafana and alert on trends instead
               of waiting for individual expiry emails.
      
            
      Setup
      
            
      Enable the endpoint and copy the scrape token. In Prometheus, add a scrape job
               with authorization: credentials: <token> (or the equivalent
               bearer-token config in your agent). The token only grants /metrics
               — nothing else.
      
            
      Gotchas
      
            
      Metrics are aggregate by design (no per-certificate labels that would explode
               cardinality or leak inventory into your metrics store). Alert on
               expiry-bucket growth and delivery failures first — those are
               the two that page-worthy incidents grow from.
      
          
      

          
      
            
      Admin · Automation API tokens Commercial / Government
      
            
      Bearer tokens for headless callers — Terraform, Ansible, CI pipelines,
               cert-manager — against the automation API (POST /api/v1/certificates
               and its status endpoint). Machines get scoped tokens instead of borrowing a
               human's session.
      
            
      Setup
      
            
      Create a token per integration (not per team) and scope it to the templates it
               may request against. The secret is shown once; store it in your secret manager.
               Every token's requests still flow through the same issuance policy and audit
               trail as interactive requests.
      
            
      Gotchas
      
            
      Scope tightly: a token limited to one template can't be repurposed quietly.
               Rotate by creating the replacement first, cutting the caller over, then revoking
               the old one — revocation is immediate.
      
          
      

          
      
            
      Admin · Crypto posture — CBOM & PQC readiness Commercial / Government
      
            
      A Crypto Bill of Materials over your certificate inventory: the algorithm mix,
               quantum-vulnerable and deprecated-crypto counts, and — most usefully — the list
               of certificates to rotate FIRST. Export a CycloneDX CBOM for your SBOM/compliance
               tooling.
      
            
      Reading it
      
            
      The posture cards summarize what your estate runs on (RSA sizes, EC curves,
               signature algorithms). "Quantum-vulnerable" counts classical public-key certs —
               which today is most of them; the point is the trend and the rotation order, not
               the raw number. Weak/deprecated items (small keys, SHA-1) are the immediate
               to-do list.
      
            
      Gotchas
      
            
      The CBOM covers what Certheim can see — grow the fleet scan/inventory first and
               the CBOM gets truthful with it. PQC readiness is a journey: use the export to
               brief security leadership, then track the vulnerable count DOWN release by
               release.
      
          
      

          
      
            
      Admin · SSH certificate authority Commercial / Government
      
            
      Sign user and host SSH certificates from a single CA key, so your fleet trusts
               the CA instead of sprawling authorized_keys and
               known_hosts files. Issued certs are short-lived, inventoried, and
               revocable via a key revocation list (KRL).
      
            
      Setup
      
            
      Generate (or import) the SSH CA key, then: on servers, set
               TrustedUserCAKeys to the CA public key to accept user certs; on
               clients, add @cert-authority to known_hosts to accept
               host certs. Issue certificates from this page (or automation) with principals
               and TTLs that match your access model.
      
            
      Gotchas
      
            
      Short TTLs are the security win — prefer hours-to-days and reissue, keeping
               revocation exceptional. When you must revoke, publish the updated KRL to servers
               (RevokedKeys); a KRL nobody distributes revokes nothing.
      
          
      

          
      
            
      Admin · DNSSEC key rollover (NetScaler) Commercial / Government
      
            
      Citrix NetScaler/ADC signs DNSSEC zones but never rolls its own keys — left
               alone, signatures age out and the zone collapses into SERVFAIL for every
               validating resolver. Certheim automates KSK/ZSK rollover over the NITRO API
               using the RFC 6781 pre-publish / double-signature methods, on schedule.
      
            
      Setup
      
            
      Add the NetScaler with its NITRO endpoint (credentials come from the service
               environment, not the database), select the signed zones, and set the rollover
               cadence. Certheim tracks each zone's key ages and walks the multi-step rollover
               automatically; the scheduler pass drives it, so keep the timer healthy (the
               Service overview health check watches this).
      
            
      Gotchas
      
            
      KSK rollovers involve the parent zone: the new DS record must reach the
               registrar/parent before the old KSK retires — the rollover pauses at that step
               until the DS is live. Roll ZSKs freely; treat KSK cadence as a planned,
               registrar-aware event.

      Admin · Database
      
            
      Certheim runs on SQLite (the default — single node, zero setup,
               one file to back up) or PostgreSQL (for high availability and
               Kubernetes, where several replicas or a managed database make sense). This page
               shows which backend is active and its health; the choice itself is made by the
               CERTHEIM_DB_URL environment variable, not from the UI.
      
            
      Staying on SQLite
      
            
      Nothing to do. The database lives in the data directory; back it up like any
               file. Watch free disk space — SQLite fails writes long before the volume is
               actually full (the health check on the Service overview page warns early).
      
            
      Moving to PostgreSQL
      
            
      Provision an empty database and a role, set CERTHEIM_DB_URL
               (e.g. postgresql://certheim:…@db.example.com/certheim) in the
               service environment, run the bundled migrate step, and restart. Migration is a
               one-time copy; the SQLite file stays behind as a fallback snapshot of that
               moment. Credentials live in the environment file only — never in the UI or the
               database itself.
      
            
      Gotchas
      
            
      Point several app replicas at the SAME PostgreSQL database only — two instances
               on separate SQLite files silently diverge. After switching backends, re-run one
               manual scheduler pass (Service overview → health check) to confirm timers write
               to the new store.
      
          
      

          
      
            
      Admin · Trust store
      
            
      Build your organization's CA trust bundle inside Certheim instead of hand-editing
               trust anchors over SSH. Upload root and intermediate certificates, and Certheim
               maintains a single ordered bundle you can install locally, push to fleet hosts,
               or download for other tooling.
      
            
      Building the bundle
      
            
      Upload each CA certificate (PEM). Certheim validates that each file is a CA
               certificate, dedupes by fingerprint, and keeps roots and intermediates
               distinguishable. Remove an anchor and the bundle is rebuilt immediately —
               distribution targets pick up the change on their next install/push.
      
            
      Distributing it
      
            
      Install on this host drops the bundle into the OS trust store via the
               privileged helper (VM installs). Push over SSH sends it to fleet hosts
               using a delivery credential. Container installs typically mount the downloaded
               bundle or bake it into the image instead.
      
            
      Gotchas
      
            
      The bundle is trust ANCHORS only — leaf certificates don't belong here and are
               rejected. Rotating your internal root? Upload the new root alongside the old
               one, distribute, and remove the old root only after every consumer trusts the
               new chain.

      Government · Access control
      
            
      Separation of duties enforces that whoever requests a
               certificate cannot approve or sign their own request — a different
               authorized user must (NIST AC-5). Violations are blocked at approval time with a
               clear reason.
      
            
      Dual control (four-eyes) additionally requires
               M-of-N approvals before sensitive actions — issuance and
               revocation — take effect: the action holds pending until the quorum approves
               (NIST AC-6).
